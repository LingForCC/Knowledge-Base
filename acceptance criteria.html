Acceptance Criteria

Acceptance Criteria is the final output of requirement refinement. It leverages the concept of state machine to describe the behaviors of a software application. It is a bridge between requirement and technical design/test strategy. And this is also the most important thing engineering team need to confirm before starting to do the actual implementaiton.

 

Format

Theoretically, there isn't a dedicated format when writing Acceptance Criteria. But personally I recommend following the format introduced by Dan North as described below(more info can be found in Wiki.

Scenario 1: Refunded items should be returned to stock
Given that a customer previously bought a black sweater from me
And I have three black sweaters in stock
When he returns the black sweater for a refund
Then I should have four black sweaters in stock

Scenario 2: Replaced items should be returned to stock
Given that a customer previously bought a blue garment from me
And I have two blue garments in stock
And three black garments in stock
When he returns the blue garment for a replacement in black
Then I should have three blue garments in stock
And two black garments in stock

There are several advantages when using this format

1. Understand Technical Dependency
Usually, a well-designed software is constructed by a bunch of functional modules. Based on the Given phrase, it is easy to figure out the technical dependency between existing modules and current to-be-implemented module. Take Scenario 1 mentioned above as an example, based on the content in Given phrase, it is easy to figure out that the to-be-implemented module needs to interact with two existing modules. One is the module which tracks what user bought before. And the other one is the module which tracks the inventory.

2. Estimate Impact Range
Once the dependency is identified, developers can use it to estimate the impact range, which means developers can understand the feature described in this Acceptance Critieria could be impacted when changes are applied to related modules in the future. In this way, developers can provide an estimation of the things need to be tested after each code change.

3. Design Unit Test and Integration Test
Besides, the dependency also helps developers to understand how to do Unit Test or Integration Test for current Acceptance Criteria, since the related modules are the things need to be mocked.

 

Testable

AC should be testable. An example as below is not testable.

Given the click to dial is enabled
Then the app should detect the phone numbers on web pages

Because, it is impossible to test it against all the web pages in the world. So the correct way to do this is to list the pages we need to support specificly.

 

Ubiquitous

AC should be ubiquitous, which means that everyone involved in the writing should have the same understanding. Most of the time, people will assume that others have the same understanding with them when reading the same thing. So it is always good to list all the potential points which could make different understandings.

As the example below

When user input Non integer number
Then user will be prompt warning

It looks quite simple. But actually, there could be different understanding when testing against Non integer numbers. It is good to list Non integer number as examples in this case.

When user input
Then user will be prompt number format warning
Example:
| integer |
| 0.1 |
| abc |
| -43 |
| 3@a |
